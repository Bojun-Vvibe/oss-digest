The most consequential is **#10254** ("fix: use deterministic keys for MCP server tool routing"), which targets issue #8087. MCP tool function names currently embed a server key generated via `nanoid(5)`, so after an extension restart or MCP reconnect the key changes while the model still has the old tool name cached in context — producing "No connection found for server" routing failures. The PR replaces the random nanoid with a deterministic djb2 hash of the server name, so the same server always resolves to the same key across sessions. If you maintain or rely on MCP integrations, this is the one to read.
