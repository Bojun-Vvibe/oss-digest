## v0.46.1 ships major file-tool + LSP performance fixes

The headline today is the **v0.46.1 release**, which addresses long-standing complaints that `view` and `edit` tool calls could stall for seconds at a time. The root cause: Crush notifies LSP servers about touched files and waits for replies, and slow LSPs were dragging tool latency with them. The release notes explicitly call out a new 300ms cap so a sluggish LSP can no longer block your session.

The fixes all landed via **PR #2317** ("fix(lsp): slow view + fix lsp find references", merged by @meowgorithm with co-author @taigrr). Worth reading if you've been frustrated by tool latency, since the changes are broader than the title suggests:

- Stop reading entire files just to count lines (commit `12f0d9f`)
- Stop watching files for changes when only reading them (`6604dd0`)
- Brief pause for LSP diagnostics on view, with a single shared timeout for parallel diagnostics collection (`3c5fcff`, `d98c854`)
- Drop the `view`-tool LSP timeout from 5s → 300ms
- Fix the LSP **Find References** tool, which was broken (`02ec5db`)
- Fix `view` tool paging plus added edge-case tests (`0583221`)

If you maintain integrations or care about the concurrency model, `d3682ac` ("use new wg pattern") is a small but relevant refactor of the waitgroup usage backing these timeouts.

The other merge of the day is **PR #2315** by @aisk, adding a **MiniMax (China) provider**. It's a straightforward provider addition mirroring the corresponding catwalk PR (charmbracelet/catwalk#191) — interesting if you're tracking provider coverage or operate in CN, otherwise skippable.

No new issues, no open PRs, and no other notable activity in the window. Net-net: upgrade to v0.46.1 if you use LSP-backed languages, and skim #2317 if you want to understand the new latency budget around file tools.
