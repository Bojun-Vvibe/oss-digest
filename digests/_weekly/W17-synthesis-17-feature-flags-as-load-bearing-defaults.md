# W17 cross-repo synthesis #17 — feature-flags-as-load-bearing-defaults: when a default is a contract (2026-04-24T15:30Z)

## The pattern

Every feature flag eventually picks a default. The default is the value the user gets when they have not (yet) opted in or out, which means the default is the *de facto* contract for the overwhelming majority of users — most users never visit the settings panel that the flag lives in. Because the default is doing all the actual work in the wild, **a change to the default is functionally identical to a breaking change to the contract**, even if no API surface moved.

This week, the four tracked repos generated a tight cluster of incidents that all share the same root: a flag was added or its default was nudged, and the change was treated as a UX detail rather than a contract change. The shape recurs in three sub-flavors:

1. **A flag is added to a feature whose existing behavior is now reachable only via the non-default value** — the old behavior becomes a "preference" that the user has to discover.
2. **A flag is added because the *advertised* default does not match what the runtime actually does** — the flag is a band-aid for a contract drift the team would rather not fix at the spec level.
3. **A flag's default is the only thing standing between two user populations that want incompatible behaviors** — picking the value is irreducibly political.

All three are visible in W17. They share a structural failure mode that is distinct from the synthesis #14 "registry drift" pattern (which is about *static* config drifting from the *running* registry) and from synthesis #16 "accepted-but-unpropagated" (which is about a *user-initiated* mutation not reaching live consumers). Synthesis #17 is about the *team* pretending a default is neutral when it is in fact the contract.

## Concrete instances from W17

1. **`use_chat_completions_api` lands as a per-model boolean ([BerriAI/litellm#25346], merged 2026-04-24T15:25Z after 16 days open).** The motivation is real: when a user points the `openai/` provider at a custom `api_base` (llama.cpp, vLLM, or any other OpenAI-compatible server that does not yet implement `/responses`), the proxy needs to bridge `/responses` requests back to `/chat/completions`. Before this PR, the bridge was implicit and based on provider-name heuristics; after it, the bridge is a *flag*. The contract change is subtle: the same provider name (`openai/`) now describes two different *protocols* depending on the flag value. Whether the request goes out as `/responses` or as `/chat/completions` — a difference that determines tool-call format, streaming shape, and reasoning-token surface — is now controlled by a per-model boolean whose default depends on whether the user remembered to set `api_base`. The default is doing all the work; flipping it (or forgetting to set it) silently changes the wire protocol.

2. **`compaction prune by default` opened then closed without merge ([anomalyco/opencode#24127], opened 2026-04-24T08:41Z, closed 14:44Z).** The PR proposed flipping the `prune_on_compact` default from `false` to `true`, which would have substantially reduced session sizes for active users. It was closed because the closing maintainer noticed that pruning compacted history broke session-replay for users who had paused mid-session and resumed days later. The PR author treated the default as a tunable; the maintainer treated it as a contract. The decision-by-PR-comment is exactly the right outcome — defaults *are* contracts and changing them needs the same review bar as a function signature change — but the very fact that the PR got to "ready for merge" before the contract issue was raised means the team's PR template does not currently include "did you change a default?" as a checklist item. (Compare: any sensible release-engineering template asks "did you change a public API?" by default.)

3. **`full access permissions lower than auto-review` ([openai/codex#19384], filed 2026-04-24T14:56Z).** The user reports that operations that succeed under `auto-review` mode silently fail under `Full Access` mode, on the same Windows machine, against the same repository. The runtime semantics of the two modes are not what the names suggest: `auto-review` apparently includes an escalation-and-confirm path that ends up granting permissions the static `Full Access` profile does not include. From the user's perspective, the *most permissive* default is *less permissive* than the second-most-permissive default. This is a mode-name promise that the runtime does not keep. It is structurally adjacent to synthesis #11 (presentation lies) but the failure is *named-default lies*: the user picked the option whose name promises the most, and got less.

4. **`gpt-5.5` is the recommended Codex model and it does not support native compaction ([openai/codex#19386], filed 2026-04-24T15:26Z).** The model is positioned as the recommended Codex model. The advertised context window is 400k tokens. The actual recoverable context window — past which compaction fails and the session becomes unrecoverable — is roughly 220k. The default is not a flag here, it is the marketing surface, and the runtime-vs-marketing gap is ~45%. The reporter's request is functionally a request for a *flag*: "let me opt out of `gpt-5.5` and use `gpt-5.4` for long sessions, because the default behavior of the recommended model is broken for the recommended workload." That request will likely be granted as a config knob (`disable_gpt_5_5_for_long_sessions = true` or equivalent), and the moment that knob lands, it becomes load-bearing for every user who runs into the 220k boundary — which is to say, every user who runs the recommended model on the recommended workload.

5. **`enable_compact_for_5_5` shape: `[openai/codex#19009]` and `[openai/codex#19329]`.** Both are open issues from earlier in the week reporting that auto-compact is hitting recurring errors. The team's likely fix is, again, a flag that disables auto-compact for affected models — which becomes a load-bearing default the day it lands.

6. **MCP-server merge skipped on custom-provider path ([openai/codex#19363], opened 2026-04-24T12:21Z).** The custom-provider code path bypasses the MCP-server-merge step that the default-provider path runs. The two paths *should* be the same path; they aren't, because the custom-provider path was added as an opt-in feature flag and the merge step was added later only on the default-provider path. The flag (custom provider on/off) is now load-bearing for whether MCP servers are honored at all in that session — a surface the user did not expect to be coupled to provider selection.

7. **`prune_excess_images` default discussion ([charmbracelet/crush#2613], comments 2026-04-24T15:21Z).** The PR has been argued for two days about what the *default value* of the new `image_history_limit` knob should be. `3` discards the fourth image silently and breaks image-heavy users; `30` reproduces the original session-deadlock for the most aggressive users. The maintainer's compromise — default `3` plus a TUI warning when truncation happens — is a tactical fix, but it concedes the point: there is no value of `image_history_limit` that does not break some user's expectation, because the *prior* default was "no limit" and any non-zero limit changes the contract. The comment thread reads like a small-scale version of the synthesis #11 cluster: any default the team picks is wrong for someone, and the only path that doesn't break someone is to keep the prior default and require explicit opt-in for the new behavior.

8. **`OpenAI-curated plugin manifests are repeatedly validated during normal turn/subagent setup` ([openai/codex#19382], filed 2026-04-24T14:47Z).** The default is that the `~/.codex/.tmp/plugins` cache is full-validated on every turn for every plugin in the curated marketplace, regardless of whether the plugin is enabled. The user's plugin config explicitly sets `enabled = false` for the affected plugin; the cache loader does not respect that. The default behavior of the cache loader assumes "if it's in the cache, validate it"; the user's reasonable expectation is "if it's not enabled, don't touch it." Pairs with a previously-merged fix (referenced in the issue as #17638) that addressed *one* code path of the same shape; the *default* policy of the loader was never changed, so a second code path with the same loader policy still hits the same surface.

9. **`auth_exception_handler` ACL surface ([BerriAI/litellm#26421], opened 2026-04-24T12:17Z).** The default for `GET /v1/models` is to return the globally-loaded model list, ignoring the per-user ACL. This default is older than the per-user-ACL feature; the ACL was added later and was wired into the chat-completions path but not the listing path. The fix is one line; the reason the bug is a year old is that the default policy of "list endpoints return the global catalog" was never reconciled with the contract that "ACLs apply to everything the user can see."

## Why these incidents are the same shape

Each instance has the same structure:
- A behavior is reachable in two ways (default and non-default, or marketed-default and runtime-default).
- The default carries the contract; the non-default is a "preference."
- The team's mental model treats the default as a *tunable* — something you can change in a minor release with a release note.
- The user's mental model treats the default as the *spec* — what they will get if they don't read the release notes.
- When the two mental models diverge, the user-visible result is a "regression" that the team can correctly point at as "we documented the change in release-notes section 4.3."

The failure mode is not a bug in the flag system. It is a bug in the team's understanding of what defaults *are*. A default is a promise that the team will not change without warning. Any test you have for the prior default value is, retroactively, a test that the contract is unchanged — and any change to that default needs to break that test on purpose, with review.

## What a "defaults-are-contracts" discipline looks like

The clearest-cut instance this week — opencode's #24127 — shows what the discipline looks like *informally*: a maintainer recognized that flipping a default could break session-replay for paused users, and closed the PR. That is the right outcome. To make it the *systematic* outcome, the four tracked repos would need:

1. **A PR-template question that asks "did this PR change a default?"** with the same prominence as "did this PR add a public API?"
2. **A test for every default that asserts the value, not just the behavior.** (`assert default_compaction_prune_on is False` rather than `assert behavior_under_default_compaction_prune_off()`.) That gives the reviewer a one-line diff that flags the contract change; without it, a default change can land in a 200-line diff and be invisible.
3. **Major/minor release semantics that treat default changes as breaking changes.** litellm and opencode both ship daily; codex ships weekly; crush ships per-PR. None of them treat a default-value change as requiring a major-version bump. (Compare: changing the default value of a `--flag` in a Unix CLI is universally treated as a backward-incompatible change.)
4. **An explicit policy for "marketing default" vs "runtime default" gaps**, of which #19386 is the worst case this week. When a model is marketed as having a 400k context window and the runtime cannot recover sessions past 220k, the team owes either (a) a runtime fix that closes the gap or (b) a marketing change that doesn't promise the un-deliverable. A flag that lets the user opt out of the broken default is a *third* path that the team should be explicit about taking, not a default path.

- **Synthesis #16 (accepted-but-unpropagated)** is about *user* mutations not reaching live consumers. #17 is about *team* changes (default-flips) that the user never explicitly accepted. They are mirror-image patterns: the same propagation gap, in the opposite direction.

## Net read

W17's cluster of default-related incidents — `use_chat_completions_api`, `prune_on_compact`, `Full Access` vs `auto-review`, `gpt-5.5` recommended-model, `image_history_limit`, marketplace-plugin auto-validation — reads as a single message: **the four projects have outgrown the era when a feature flag was a personal preference**. At today's scale, every default is the contract for tens of thousands of installs that will never read the release notes, and changing one is the same kind of action as renaming a public function. The teams that develop a "defaults-are-contracts" reflex first will spend the next quarter shipping fewer regressions; the teams that don't will spend it writing release-notes that no-one reads.

The single most actionable change available to all four projects this week is to add the PR-template question. It is one line in a markdown file. It catches every incident in this synthesis with a five-second reviewer prompt. It has no runtime cost. It is the cheapest possible high-leverage change to engineering process, and the only thing that has stopped any of the four projects from adopting it is that defaults have not, until very recently, been load-bearing enough to notice.
