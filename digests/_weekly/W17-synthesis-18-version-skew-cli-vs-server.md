## The pattern

Every agent product in the W17 set has, at its core, a *split* topology: a local CLI / desktop / extension component talks to one or more remote services (the model API, the proxy, an MCP server, an app-server, a cloud-side compactor). That split is always advertised as transparent — install the latest CLI, point it at the service, and it just works. In practice, the seam is never transparent. **Every release of the local component implicitly assumes a band of compatible remote-side versions, and every release of the remote service implicitly assumes a band of compatible local versions.** When the assumed band is *also* never written down, drift is inevitable, regressions arrive in pairs (one from each side), and the user-visible failure is some bizarre middle-state that neither side can reproduce in isolation.

Synthesis #18 names this pattern: **version-skew between the CLI/local layer and the server/remote layer is a load-bearing assumption that nobody documents until it breaks**, and when it breaks it breaks in a way that the user experiences as a flaky bug, the local team experiences as "works on the latest server," and the remote team experiences as "works on the latest CLI."

This week the pattern shows up in five distinct forms across the four tracked repos:

1. **CLI-side regressions that the server-side never saw**, because the CLI changed how it speaks to the server (#19330, WebSocket fallback).
2. **Server-side responses that the CLI cannot parse**, because the server's response shape changed and the CLI was not updated (#19386, native compaction surface).
3. **Local-cache state that survives a server-side change**, because the cache is keyed on a now-stale assumption (#19300, Remote SSH Desktop reuses stale app-server).
4. **Local-binary metadata that the OS rejects** because the signing pipeline shipped without checking what the local launcher path actually requires (#19335, AMFI keychain entitlements).
5. **CLI-side configuration that the server side ignores**, because the CLI never sends it on the wire (#19185, `config.toml` context-window).

Every one of these is a version-skew bug. None of them is documented as such; each is filed, individually, as a regression against a single layer.

## Concrete instances from W17

1. **`Codex CLI 0.123/0.124 regression: WebSocket fallback then stream disconnected before completion on Windows` ([openai/codex#19330], filed 2026-04-24T09:17Z, updated 15:25Z).** The user reports that the same Windows machine, terminal, and network was stable on `0.122.x` and broke on `0.123.x`. The CLI now attempts WebSocket transport first, falls back to HTTPS on connection failure, and then the HTTPS fallback disconnects mid-stream with `os error 10053`. This is the cleanest possible expression of CLI-side version skew: the *transport* (a wire-format choice) was changed in a CLI minor release without a corresponding bump in any documented compatibility surface. The server-side accepts both transports — that's why the team didn't think a "server bump" was required — but the *fallback* behavior depends on a server-side timeout that does not match the CLI's new retry window. The skew is between two parts of the same logical service: the WebSocket gateway and the HTTPS gateway, with the CLI assuming they fail in compatible ways. They don't.

2. **`GPT-5.5 Codex session hits unrecoverable compaction failure around ~220k tokens despite advertised 400k context` ([openai/codex#19386], filed 2026-04-24T15:26Z).** This is a server-side surface change: the model `gpt-5.5` is served by an upstream that does *not* implement `/v1/responses/compact` (the native compaction endpoint the CLI relies on for long-running sessions). The CLI assumes that any model marked as a Codex model supports the compaction endpoint; the server has shipped a Codex model that does not. The user's session works perfectly until ~220k tokens, at which point the compaction request errors with `invalid_request_error` ("请求格式非法") or `503 Service Unavailable`, and the session becomes unrecoverable because the prior history is too large to fit in a non-compacted next turn. The CLI cannot tell, in advance, that this particular Codex model does not support compaction. There is no version negotiation between the two sides; there is only a static assumption ("Codex models support compaction") that the server has silently violated.

3. **`Remote SSH Desktop reuses stale app-server and masks model_not_found as Reconnecting` ([openai/codex#19300], filed 2026-04-24T06:18Z, updated 15:13Z).** The desktop app caches a long-lived connection to an app-server. When the user updates the desktop, the new desktop reuses the cached connection to a *prior-version* app-server. The prior-version app-server returns `model_not_found` for models that the new desktop knows about (because the model catalog moved on the server side), but the desktop interprets the error as a transient network problem and shows "Reconnecting." The user sees an infinite reconnect loop with no actionable error. The skew is *between two versions of the same logical client*, mediated by the app-server cache, and the failure is masked by a presentation layer that conflates "cannot reach server" with "server says no."

4. **`macOS npm alpha 0.125.0-alpha.1 is killed by AMFI due to new keychain entitlements` ([openai/codex#19335], filed 2026-04-24T09:26Z, closed 15:28Z).** The CLI binary's signed entitlements changed in `0.125.0-alpha.1` to add `com.apple.application-identifier`, `com.apple.developer.team-identifier`, and `keychain-access-groups`. These are *restricted* entitlements that AMFI validates against a matching provisioning profile. Standalone npm-shipped binaries do not have a matching provisioning profile, so AMFI kills the process with exit 137 before any code runs — even `codex --version`. The version skew here is between the *signing pipeline* (which assumes the binary will be launched in a context that has a profile) and the *npm distribution channel* (which has never had one). The reporter's diagnosis is excellent: the binary's signature is structurally fine for a desktop-app launch path; it's actively harmful for the standalone CLI path. The fix landed in 8 hours because the diagnosis was so clean, but the *class* of bug — a signing change that assumes a launch context — will recur the moment any other entitlement is added without checking the npm path.

5. **`config.toml context window settings are not respected` ([openai/codex#19185], filed 2026-04-23T19:11Z, updated 14:57Z).** The user sets `model_auto_compact_token_limit = 180000` in `config.toml`. The schema validates. The runtime ignores it and continues to use the default. This is, on its face, an opencode-style "accepted-but-unpropagated" bug (synthesis #16), but the structural cause is different: the CLI reads the config, the CLI sends the chat request to the server, and *the server* enforces the auto-compact threshold based on its own internal knowledge of the model, not on a CLI-supplied parameter. The CLI never sends the `model_auto_compact_token_limit` field on the wire, because the server's `/v1/responses` schema does not accept it. The skew is between what the CLI's config schema *accepts* and what the wire format *carries*. Every config knob that lives on the CLI side but is enforced on the server side has this shape latent in it.

6. **`POST /model/update` does not invalidate in-process router cache ([BerriAI/litellm#26427], opened 2026-04-24T13:25Z).** Different layer of the same pattern. The proxy's REST API accepts a model update; the in-process router cache is not invalidated. From the user's perspective, the management-API version (which knows about the new model) and the inference-path version (which still serves from the cached list) are out of sync within a single proxy process. This is *intra-process* version skew: two parts of the same binary disagree about which version of the model registry is current.

7. **`Litellm clean litellm oss staging 04 01 2026` ([BerriAI/litellm#25856], open since 2026-04-16, updated 2026-04-24T14:51Z).** A nine-day-old staging-cleanup PR that has been open through the entire `merge main` flood named in synthesis #15. The PR's purpose is to reconcile the OSS staging branch with the production branch, which is itself a structural acknowledgment that the two have drifted. The branch-level version skew is being managed by a single weekly "clean" PR rather than by a continuous integration pipeline; that cadence is fine when changes are small but it is the structural reason the staging branch needs the seven `merge main` rounds named in the addendum.

8. **`merge main` PR cluster ([BerriAI/litellm#26430, #26431, #26432, #26433, #26437], all opened and merged within seconds, between 14:29Z and 15:23Z).** Synthesis #15 named this from a sync-debt angle. From a version-skew angle, the same data says: **the staging branch and the main branch are in a continuous reconciliation, sub-minute cadence, and the diff between them is what defines "what version of LiteLLM users will get" at any given moment.** A user pinning a `litellm` version pulled from `main` in the morning may, by the afternoon, be running effectively-the-same code as a user pinning a `litellm` version pulled from `litellm_internal_staging` — but the *intermediate* moments, when only one side has absorbed a change, are skew windows. Anyone running `litellm` in production needs to know which branch they pinned to, and the answer "they're identical now" is true only at merge-bottom, not in general.

9. **`Codex hooks landed on Windows` ([openai/codex#17478], closed 2026-04-24T15:26Z, originally 2026-04-11).** A platform-capability that lived on macOS/Linux for two weeks before reaching Windows. During that gap, any documentation that said "Codex hooks let you do X" was correct on two of three platforms and wrong on the third. The version skew here is between the *documentation* (which assumed a single Codex platform) and the *implementation* (which had three). The fix landing closes that skew, but the next platform-asymmetric capability will reproduce it.

## Why version skew is its own pattern

The instinct, reading the issues above, is to file them under a half-dozen pre-existing categories: `19330` is a CLI bug, `19386` is a model bug, `19300` is a connection-state bug, `19335` is a signing bug, `19185` is a config bug, `26427` is a cache bug. Each filing is correct. Each is also incomplete, because every one of them is a manifestation of the same structural issue: **two halves of a split system shipped on independent release cadences, with no protocol-level negotiation, and the assumption that they would remain compatible was implicit.**

The fix-shape is genuinely different from the other syntheses:
- **Synthesis #11 (presentation/enforcement)** wants two layers in the same process to read from one source.
- **Synthesis #14 (registry drift)** wants the static catalog to match the live runtime *within a single component*.
- **Synthesis #16 (accepted-but-unpropagated)** wants a user-initiated mutation to reach all live consumers within a single process.
- **Synthesis #17 (defaults are contracts)** wants the team to treat default-value changes with the rigor of API changes.
- **Synthesis #18 (version skew)** wants the CLI and the server to *negotiate* their compatible band at connection time, with a documented failure mode when they don't agree.

The first four are intra-component or intra-team disciplines. The fifth is genuinely cross-component, and it requires a *protocol-level* artifact that none of the four projects currently has: a version handshake.

## What a version-handshake discipline would look like

The minimum viable version handshake is a single field exchanged at the start of every session: the local component sends its version and the set of capabilities it expects, the server sends its version and the set of capabilities it offers, and either side can reject the connection with a *user-actionable* error if the bands don't overlap. None of the four projects has this:

- **codex CLI does not advertise its version to the server in any way the server can use to gate response shape.** The server returns the same shape to `0.122.x` and `0.124.x`; the CLI is responsible for parsing both shapes. When the server's shape changes (#19330's WebSocket-vs-HTTPS), older CLIs fail in opaque ways.
- **opencode does not negotiate registry-version compatibility with its provider transformer layer.** When the registry catalog moves (synthesis #14), the transformer keeps using the prior catalog until restart.
- **litellm does not version its `Router` cache state.** When the management API mutates the registry (#26427), the in-process router has no way to know its cache is stale.
- **crush has no documented compatibility contract between its agent CLI and the model providers it calls.** Provider-side response shape changes (e.g. tool-call format) propagate as `unmarshal` errors.

A protocol-level version handshake would not solve every bug listed above — #19386's "advertised 400k, recoverable 220k" is a contract-shape bug that no handshake catches, because both sides agree the model exists. But it would convert the *failure mode* of the others from "infinite retry / silent regression / opaque error" into "this CLI is too old / too new for this server, please upgrade / downgrade." That is a strictly user-actionable improvement.

## Net read

The four tracked projects have collectively spent W17 absorbing the cost of *not* having a version-handshake discipline. The cost shows up as a steady stream of issues that the individual layer's maintainer reads as "weird" or "intermittent" or "Windows-only" or "specific to this model" — and the structural reading is "the two halves of the system disagree about what version they are talking to, and there is no documented seam at which they would notice." Until at least one of the four projects (the obvious candidate is litellm, since its proxy already mediates a multi-version surface) ships a documented version-handshake, this synthesis will keep producing fresh instances every week.

The minimum useful change is small. The *adoption* cost is the high one, because every existing client and server has to learn to send and respect the field, and every existing deployment has to be tolerant of clients and servers that don't yet send it. That is exactly the cost that version-handshakes pay for themselves: once the field exists and is non-empty on both sides, every future protocol change can be gated on it, which means the *next* WebSocket-fallback regression can announce itself in a way the user can act on rather than as an `os error 10053` mid-stream.
