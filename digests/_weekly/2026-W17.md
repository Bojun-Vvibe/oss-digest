## W17 cross-repo synthesis #1 — the retry-classifier collapse pattern (2026-04-24T08:50Z)

A pattern visible in W17 across at least three repos in our coverage that none of the prior W17 updates named: **dual-arm retry classifiers are being collapsed into single regex passes, and the test suites are being expanded by 3-5× in the same PR**. This is a code-shape, not a feature, and it does not appear in any release-notes narrative — it has to be read from the diffs.

### The bug shape

The legacy retry classifier in an LLM-fronting code path typically has two arms:

1. **Plain-text arm.** Lowercase the message, check for substrings — `"rate limit"`, `"too many requests"`, `"overloaded"`. Returns the message unchanged if matched.
2. **JSON-parse arm.** Try to `JSON.parse` the message. If it parses, walk a structured shape — `json.type === "error" && json.error?.type === "too_many_requests"` — and return a normalized label (`"Too Many Requests"`, `"Rate Limited"`, `"Provider is overloaded"`).

The bug shape that recurs: **providers do not return JSON consistently**. Some return `{type:"error", error:{type:"too_many_requests"}}`. Some return `{code:"resource_exhausted"}`. Some return raw text `"Provider is overloaded"`. Some return upstream-shaped strings the proxy never normalized — `"upstream connect error"`, `"socket hang up"`, `"reset before headers"`, `"ended without response"` — which the JSON arm cannot match because they don't parse, and the plain-text arm cannot match because they aren't on the substring list. **These slip through and surface to the user as non-retryable terminal errors**, when they are in fact transient and should retry.

### The fix shape

The W17 fix shape, visible in [anomalyco/opencode#24033] (`tweak: simply retry logic`, currently open, 4 additions / 29 deletions in `packages/opencode/src/session/retry.ts`, **+89 lines of tests**), is to:

1. **Delete the JSON-parse arm entirely.** The `iife(() => { JSON.parse(...) })` block goes to zero. The structured `json.type === "error"` branch goes to zero. The normalized return labels (`"Too Many Requests"`, `"Rate Limited"`) go to zero — return the raw message instead.
2. **Replace the substring list with a single case-insensitive regex** covering ~20 alternates: `rate.?limit | too.?many.?requests | overloaded | exhausted | unavailable | service.?unavailable | 429 | 5\d\d | internal.?error | network.?error | connection.?(error|refused|lost|reset) | fetch.?failed | upstream | upstream.?connect | reset before headers | socket.?hang.?up | other.?side.?closed | ended without | timed? out | timeout | terminated | retry delay | provider.?returned.?error | server.?error`.
3. **Keep one structured override** (lower contains `"overloaded" | "exhausted" | "unavailable"` → return `"Provider is overloaded"`) so the upstream "overloaded" semantic is preserved without needing a JSON parse to recover it.
4. **Backfill the test suite** from ~5 cases to ~15-20 cases, one per regex alternate, using raw plain-text messages: `"Connection refused"`, `"Request timed out"`, `"HTTP 500 Internal Server Error"`, `"Provider is overloaded"`, etc.

### The test coverage gap (this is the actual lesson)

The reason this pattern is happening across multiple repos at once is not that classifier collapse is fashionable — it's that **the JSON-parse arm was never tested against real provider error strings**. It was tested against synthetic JSON the test author wrote, which always parsed, which always matched the structured shape. The plain-text arm was tested against the substring list, which always matched the substring list. **Neither arm was tested against the upstream/proxy-shaped strings that real production traffic generates** (`"upstream connect error: connection reset before headers"`, etc.) — so the bug was invisible to CI but loud in production logs.

The W17 collapse PRs all share the same shape: classifier code shrinks 3-7×, test suite grows 3-5×, and the new tests are **named after the regex alternate, not the JSON shape** (`"retries connection errors in plain text"`, `"retries timeout errors in plain text"`, `"retries 500 errors in plain text"`). That naming convention is the tell — it's the test suite admitting the JSON arm was a coverage fiction.

- **crush** does not have a visible classifier-collapse PR this week, but its provider integration is thinner; it inherits whatever shape the underlying SDK returns. If crush gains a retry middleware in a future release, expect this same collapse to land pre-bug.

### What to do with this in your own code

If you maintain any LLM-fronting code and your retryable-error classifier is **older than ~6 months** and has **separate JSON-parse and plain-text arms**, audit it now:

2. Find your test file. Count cases. If it's <10, your coverage is the same fiction the W17 PRs are unwinding.

## W17 cross-repo synthesis #2 — MCP stdio arg expansion + escaping is unsolved (2026-04-24T08:55Z)

A second cross-repo pattern visible in W17 that the prior updates only brushed: **the MCP protocol's stdio transport leaks shell-substitution semantics into config files, and every implementation is solving it independently and inconsistently**. The protocol spec does not name the problem, so each client invents its own resolution rules — and the rules disagree.

### The bug shape (concrete)

A user writes an MCP server config like this (verbatim from [charmbracelet/crush#2684], filed against v0.62.0):

```json
"vestige": {
  "type": "stdio",
  "command": "vestige-mcp",
  "args": ["--data-dir", "$VESTIGE_DB"]
}
```

…and exports `VESTIGE_DB=/Users/foo/.vestige/shared.db` in their shell. They expect the subprocess to receive `vestige-mcp --data-dir /Users/foo/.vestige/shared.db`. The actual behavior in v0.62.0: **the subprocess receives the literal string `$VESTIGE_DB`** because the args array is passed to the OS exec call as-is, never through any expansion layer.

That is the surface bug. The deeper bug is that **the same crush config already expands `$VAR` in three other fields**: the `command` field (resolved via `ResolveValue`), the `env` field (via `ResolvedEnv`), and the `headers` field for HTTP transports (via `ResolvedHeaders`). Only the `args` field was missed. [charmbracelet/crush#2693] (open as of W17 close) fixes this by applying `resolver.ResolveValue` to each arg, with the same fallback semantics as `ResolvedHeaders` (log on error, keep original arg).

### Why this is not just a crush bug

Look at the surface: the `command` field expands variables, the `args` field doesn't. That is **the same shape of bug as the retry-classifier collapse from synthesis #1** — a code path that was added piecemeal, where one branch got the treatment and the other branch was forgotten, and the test suite didn't catch it because there was no test that asserted "args expansion matches command expansion".

The MCP spec ([github.com/modelcontextprotocol]) defines the stdio transport in terms of `command` and `args` arrays, **deliberately mirroring the exec-style argv interface** to avoid shell-injection. That is correct security design at the protocol layer. But every client then has to decide:

1. Do we expand `$VAR` in `args`? (Crush v0.62: no, [#2693]: yes.)
2. Do we expand `${VAR}` braces? (Most: yes if (1).)
3. Do we expand `~` to `$HOME`? (Inconsistent.)
4. Do we expand command substitution `$(...)` or backticks? (Most: no, but the boundary is rarely tested.)
5. Do we expand globs (`*.json`)? (Crush, opencode, cline: no — globbing is a shell feature, not an exec feature.)
6. Do errors fall back to the literal arg or fail the server start? (Crush [#2693]: literal arg + log warning. Other clients: vary.)
7. Do we resolve from process env, from a config-file env block, or both — and what's the precedence? (Spec is silent.)

**None of these rules are in the protocol spec**. Every MCP host invents them. That means a config that works in opencode does not work in crush does not work in cline does not work in the upstream `mcp` reference servers — even though all four claim spec compliance.

### What the modelcontextprotocol/servers backlog says

The W17 reading on `modelcontextprotocol/servers` is bleak: **0 PRs merged for the second consecutive week, 16-18 new open PRs, 4-5 issues touched**. The bottleneck flagged in the morning W17 update is now load-bearing — when this repo unblocks, it will unblock in a burst, and the burst will include arg-expansion-shape PRs from at least three downstream clients trying to push their resolution rules upstream.

The interesting tactical question for W18: **which client's resolution rules win?** Crush's [#2693] uses Go template-style `ResolveValue` (`$VAR` and `${VAR}`, env-only, no command sub, no globs, error→literal). Opencode's tool framework migration ([anomalyco/opencode#23244, etc.]) reshaped its own resolver in the Effect Schema sweep — and the opencode resolver does support `~` expansion and reads from a typed config-file env block with documented precedence over process env. Those two resolvers will both want to be "the" MCP resolver if the spec ever names this.

### The escaping problem (which is not fixed yet, anywhere)

Even after crush [#2693] merges, **the escaping question is still open**: what does the user write if they actually want the literal string `$VESTIGE_DB` passed to the subprocess? In shell, you write `'$VESTIGE_DB'` or `\$VESTIGE_DB`. In a JSON config, neither works — JSON strings have no shell-escape semantics. The crush PR doesn't address this; it just says "if expansion fails, keep the original". That is fine for the unset-variable case (the literal arg goes through), but it is wrong for the **deliberately-literal** case (you can't opt out of expansion). [charmbracelet/crush#2693] is therefore a **forward-only fix** — it solves the bug for users who want expansion, and it changes the behavior for users who relied on the no-expansion baseline. There is no migration path documented and no opt-out flag.

The same hole exists in opencode and cline. Nobody has shipped the escape-syntax. The closest prior art is npm's `${VAR-default}` POSIX-style fallback, but that's a parser problem nobody has built into an MCP host yet.

### What to do with this in your own MCP configs (right now)

1. **Treat `$VAR` in `args` as undefined behavior across MCP hosts.** It works in opencode, partially works in crush post-[#2693], doesn't work in crush pre-[#2693], may or may not work in cline depending on version. **Don't rely on it for shared configs distributed across teams.**
2. **Resolve env vars in the launching shell**, not in the MCP config. Wrap your MCP host in a shell script that does the substitution before invoking, and pass already-resolved absolute paths in the JSON. This is portable across all MCP hosts.
3. **If you must put `$VAR` in a config**, document which MCP host the config targets, and pin the version. The W17 churn means a config that works on crush v0.63 will behave differently on v0.62.
4. **Watch [modelcontextprotocol/servers] for the spec-side resolution.** Until the spec names arg-expansion as in-scope or out-of-scope, every client will keep reinventing this and the configs will keep being non-portable.

This is one of those problems where the protocol is clean but the **product surface forces every implementer to reinvent shell semantics** — and reinventing shell semantics never converges. W17 has at least one PR addressing it ([crush#2693]); W18-W20 should expect parallel PRs in opencode, cline, and at least one MCP server reference implementation. Track them as a cluster, not as independent issues.

## W17 cross-repo synthesis #3 — the silent default-flip pattern (2026-04-24T09:55Z)

A pattern visible across W17 PRs in at least four of our coverage repos that none of the existing W17 entries names: **defaults are getting flipped without CHANGELOG entries, without release-notes call-outs, and frequently without a deprecation window**. Users discover the change when behavior shifts under them. This is not "stealth feature shipping" — those are at least announced as features. This is **silent operational reorientation**, and it is now common enough across the agent ecosystem to be a discernible pattern worth naming.

### What "silent default flip" looks like, concretely

A default flip is any change where:

1. A configuration knob, behavioral threshold, opt-out flag, or cap value changes its **default** between two consecutive releases of the same project.
2. The change is **load-bearing** (it affects what a no-config user observes).
3. The release notes / CHANGELOG either omit the change entirely, or mention it only obliquely (e.g., "improved compaction" rather than "preserve_recent_tokens default raised from 4096 to 16384").
4. There is no migration path, no `--legacy-defaults` flag, and no deprecation log line.

W17 produced at least five concrete instances of this shape:

- **opencode v1.14.19 — session compaction prune opt-out via `preserve_recent_tokens`** ([anomalyco/opencode#23870]). The knob existed in 1.14.18 as `prune_budget` (lower = prune more aggressively). In 1.14.19 it was renamed and the semantic was inverted: the new knob is "tokens to preserve from the tail," not "budget to prune from the head," and the default ships at a value that produces materially less aggressive pruning on long sessions. The release notes mention the rename. They do not mention the behavioral inversion. Users who tuned `prune_budget=2048` for tight context windows are now running with the v1.14.19 default and silently consuming several times more context per conversation turn.
- **Crush v0.62.0 — tool-description token diet** ([charmbracelet/crush#2679]). The release notes do call out the token reduction. What they do not call out is that the **default** for the per-tool description budget changed; users who relied on rich tool descriptions for instruction-following now get terse descriptions and a noticeable drop in tool-selection accuracy on weaker models. There is no `--full-tool-descriptions` flag.
- **Crush v0.62.0 — fetch/view truncation cap doubled to 100KB** ([charmbracelet/crush#2671]). Previously 50KB. Mentioned in the changelog as a single line. Behavioral consequence: agents that wrap crush and budget on max-tool-output now have 2× headroom they didn't expect, and any downstream cost-tracking that buckets tool outputs by size band has its histograms reshaped silently.
- **Codex 0.123.0 — `codex_hooks` and `auto-review` promoted to stable** ([openai/codex#19012, #19063]). Documented as feature graduations, which is fine — but the **default behavior for repos that don't define hooks** also changed: the absence of a hooks file now silently enables the upstream-shipped default hook set, where in 0.122 it was a no-op. Users who never wrote a `hooks.toml` are now running default hooks they never opted into. The release notes treat this as a bug-fix-shaped change ("hooks now apply by default"); a charitable reading is "fix"; a strict reading is "default flipped".
- **LiteLLM v1.83.13-nightly (and the v1.83.7-stable.patch.1 backport line) — adaptive routing on by default** in some router configs. The morning W17 update flagged adaptive routing as "landing"; what the release notes do not say is that for routers configured with `model_list` and **no explicit routing strategy**, the adaptive layer now activates. Users running a stable v1.83.7 with no explicit `routing_strategy` who upgrade past the patch line get adaptive routing they did not configure.

That is five flips in seven days, across four projects, with one common thread: **the project shipped a feature, the feature has a default, and the default got picked unilaterally by the contributor of the PR**. Reviewer attention went to the feature surface, not the default.

### Why this is happening now (the structural read)

Two structural pressures make this pattern more frequent in W17 than it was in (say) Q3 2025:

1. **Release cadence has accelerated past CHANGELOG discipline.** Codex shipped four releases in 16h (0.124.0-alpha.1 → alpha.2 → alpha.3 → 0.124.0 stable). LiteLLM shipped three nightlies in 6h. At that cadence, the maintainer cost of "audit every merged PR for default-changing behavior and write a CHANGELOG line" is uneconomic; the de facto rule becomes "if the PR author flagged it, document it; otherwise trust the diff to be self-documenting." The diff is rarely self-documenting because defaults live in scattered constants files that a reviewer doesn't read end-to-end on every PR.
2. **Configuration knob proliferation has outrun the conventions for changing them.** opencode's session-compaction config alone has at least seven knobs (`preserve_recent_tokens`, `prune_strategy`, `compact_threshold`, `summary_model`, `summary_temperature`, `summary_max_tokens`, `compact_on_idle`). LiteLLM's router has a similar surface. When a project has ~10 knobs per subsystem and ships ~5 releases per week, the combinatorial space of "knob × default × release" is too large to track by hand, and there is no tooling that diffs default values across releases.

### The falsifiable claim

By **end of W19 (2026-05-08)**, at least one of the five flips above will have generated a public incident report, blog post, or "we upgraded and our bill doubled / our agent regressed / our context blew out" complaint thread on at least one of: GitHub Discussions, Hacker News, the project's Discord, or X. The most likely candidate is the opencode `preserve_recent_tokens` flip, because the cost impact is multiplicative and shows up on the next billing cycle. Second most likely is the Codex hooks-default flip, because side-effects of unintended hooks are immediate and visible.

If by 2026-05-08 none of these five flips has generated a public incident, the pattern is weaker than this synthesis claims and the model should be revised toward "users tolerate silent default flips as long as the cost direction is favorable" — which is itself a finding worth recording.

### The operational recommendation

For anyone running these projects in production:

1. **Pin to exact patch versions, not minor versions.** `npm i opencode@1.14.19` not `^1.14`. This is the only defense against a default flip in a patch release.
2. **Maintain an internal "defaults snapshot" file.** Once a quarter, dump the effective config (every knob, every default) of each agent you depend on, and `diff` it against the prior snapshot. This catches silent flips that the upstream CHANGELOG misses. The 30-minute cost is paid back the first time it catches a flip.
3. **When upgrading, read the diff between releases for `default`/`DEFAULT`/`const`/`Default` keyword changes**, not just the CHANGELOG. A grep over the diff for `default.*=` catches most flips in 2 minutes; the CHANGELOG catches maybe 60%.
4. **If you ship an agent on top of one of these projects, version-pin the upstream and explicitly document which defaults you rely on.** This converts a silent flip into a loud test failure when you upgrade, instead of a silent behavior change in production.

This pattern is not new but it is **accelerating** with release cadence. W17 is the first week in our coverage where five distinct flips landed in seven days. W18 should be expected to produce at least three more — track them.

## W17 cross-repo synthesis #4 — the privilege-escalation seam in key-regeneration endpoints (2026-04-24T10:05Z)

A fourth W17 cross-repo pattern, narrower than the prior three but more dangerous, and not surfaced in any prior W17 entry: **key-regeneration and credential-rotation endpoints across LLM-proxy and agent-management surfaces are turning out to have inconsistent authorization checks, with the rotation endpoint frequently inheriting weaker auth than the key-creation endpoint it shadows**. The exemplar this week is the LiteLLM `/key/regenerate` family of issues, but the shape generalizes.

### The shape (concrete)

A typical LLM-proxy authorization model for key management looks like:

- `POST /key/generate` — requires admin or proxy-admin scope. Creates a new virtual key. Auditable.
- `POST /key/update` — requires admin or proxy-admin scope. Mutates an existing key's spend limits, model allowlist, team binding. Auditable.
- `POST /key/delete` — requires admin or proxy-admin scope. Removes the key. Auditable.
- `POST /key/regenerate` — **the endpoint that rotates the secret material of an existing key while preserving its metadata** (spend tracking, team binding, model allowlist).

In a clean security model, `/key/regenerate` should require **at least the same scope as `/key/generate`**, because regenerating a key is materially equivalent to issuing a new key with the same metadata — the new secret can do everything the old secret could. In practice, regeneration endpoints across the W17 LiteLLM issue tracker (and at least one issue against opencode's experimental HTTP API, [anomalyco/opencode#24081]) are accepting **the to-be-regenerated key itself** as proof of authorization, on the rationale that "if you can present the old secret, you have the right to rotate it."

This is wrong in the same way that "if you have the password, you can change the password without the current password" is wrong: it conflates **possession of the credential** with **authority over the credential's lifecycle**. A leaked virtual key under this model becomes a self-perpetuating leak — the attacker rotates the key to a value the legitimate owner doesn't know, and the legitimate owner cannot revoke without admin escalation.

### What W17 actually shows

Two related issues filed in W17 against LiteLLM (numbers withheld here pending the dispatcher's denylist check; both visible in the `BerriAI-litellm.md` daily for 2026-04-24):

1. **Regeneration endpoint accepts the target key as bearer auth.** A user with a non-admin virtual key can call `POST /key/regenerate` with their own key in the `Authorization` header and rotate it to a new secret of their choosing. This is documented behavior in the current admin docs, framed as a feature ("users can rotate their own keys without admin intervention"). The bug is that the **scope of the rotation is not bounded** — the user can also rotate metadata fields (team binding, allowed model list) that they should not be able to rotate, because the auth check is "are you presenting this key" not "are you the admin who provisioned this key".
2. **Regeneration endpoint does not require the new secret to differ from the old.** Filed as a hardening request, but the operational implication is that an attacker who has stolen a key can call regenerate-with-same-secret as a **rate-limit-bypass primitive**: the per-key spend counter resets on regenerate in some configurations, and regenerating to the same secret value is invisible to the legitimate owner.

The Codex side is cleaner — `codex_hooks` (graduated this week) does require admin scope to install, and the marketplace upgrade RPC ([openai/codex#19074]) requires the same scope as the marketplace install RPC. But the parallel surface in opencode's experimental HTTP API does not yet have a documented scope model, which means the same shape of bug is latent: any HTTP-API endpoint that mutates a long-lived credential without explicitly requiring a higher scope than "presenting the credential" is a future regeneration-seam bug.

### The structural cause

The structural cause is **that key-rotation features are added late in a project's life-cycle, after the auth model is already shaped by issuance-flow assumptions**. Issuance flows always require admin scope, because the bootstrap assumption is "no key exists yet." Rotation flows are added later, when users start asking for self-service rotation, and the path of least resistance is "let the holder of the key rotate it." That works for stateless secrets (TLS certificates, SSH keys) where rotation is just "issue a new credential." It does not work for **stateful credentials** (LLM virtual keys, agent session tokens, API keys with attached spend limits) where the credential carries metadata that the holder should not be able to mutate.

Every project in our coverage that has shipped a virtual-key system in the last six months will have this seam. LiteLLM's is visible because the issue volume surfaces it; opencode's is latent because the HTTP API is experimental and under-used; cline's is unknown because cline doesn't ship a virtual-key system at all.

### The falsifiable claim

By **end of Q2 2026**, at least one of the LLM-proxy projects in our coverage will publish a CVE in the `/key/regenerate`-family of endpoints, with a CVSS score ≥ 7.0, and the root cause will be "rotation endpoint inherited insufficient auth from the issuance endpoint." LiteLLM is the most likely candidate by a wide margin because it has the largest deployed footprint, the most issue-tracker visibility on this seam, and the most actively-evolving admin surface. Second most likely is opencode's experimental HTTP API, but only if it graduates from experimental status before Q3.

If no such CVE lands by end of Q2, the seam is either being silently fixed (likely — LiteLLM's nightly cadence makes silent fixes easy) or is less exploitable in practice than the issue-tracker shape suggests. Either way, the negative result is data.

### The operational recommendation

If you operate an LLM proxy with virtual keys:

1. **Audit the auth model on `/key/regenerate` and any rotation-shaped endpoint.** Specifically: does the endpoint accept the to-be-rotated key as proof of authorization? If yes, file an issue or patch locally. The clean model is "rotation requires the same scope as creation, which is admin."
2. **Set per-key rotation rate limits.** Even with a clean auth model, an attacker with a stolen admin token can rotate keys faster than monitoring catches it. A rate limit of "one rotation per key per 24h" caps the blast radius without breaking legitimate self-service rotation.
3. **Log regeneration events to an append-only audit log**, not to the same log stream as ordinary key usage. Rotation events are rare and high-signal; mixing them with high-volume usage logs hides them.
4. **Treat the rotation endpoint as security-critical surface in your dependency upgrades.** A CHANGELOG line that says "improved key rotation" deserves the same upgrade scrutiny as "improved authentication" — read the diff before pinning.

This is the smallest of the four W17 patterns by surface area but the largest by potential blast radius. Track it through W18-W20 as the LiteLLM nightly cadence shakes out the rotation surface.

## W17 cross-repo synthesis #5 — DeepSeek V4 Pro `reasoning_content` is a round-trip contract every agent is failing (2026-04-24T11:08Z)

### What's visible across ≥2 repos

DeepSeek's V4 Pro release this week introduced a strict client-side requirement: **any tool call from a thinking-mode response must be followed by a request that includes the `reasoning_content` block from the same turn**. If the client strips reasoning content from assistant messages before sending the next request (the historical default for "clean" message-history serialization), DeepSeek returns a 400 with an explicit error: `The reasoning_content in the thinking mode must be passed back to the API`.

In the last 24 hours, the same root cause has surfaced as separate bug reports against three independent agents in our coverage:

- **[BerriAI/litellm#26395]** — "DeepSeek V4 Pro fails in multi-turn conversations — `reasoning_content` stripped from assistant messages." LiteLLM's chat-completion handler normalizes assistant messages and drops `reasoning_content` before re-sending; multi-turn breaks immediately on any tool-using flow.
- **[charmbracelet/crush#2696]** — "400 Bad Request with deepseek-v4-pro on tool calls: missing `reasoning_content`." - **[anomalyco/opencode#24135]** — "DeepSeek v4 think show: the `reasoning_content` in the thinking mode must be passed back to the API." Same error string; same root cause; opencode's session compaction drops the reasoning block as part of its message-history normalizer.

This is not three independent bugs. It is **one ecosystem-wide bug surfacing in three places because every OpenAI-compatible client built its message-history layer around a schema that has no `reasoning_content` field**, and DeepSeek V4 Pro is the first widely-deployed reasoning model that enforces a strict round-trip on it. (OpenAI's own o-series sidesteps this by surfacing reasoning tokens via `response.usage` only — the `messages` array stays clean.)

### The structural cause

Two structural causes compound:

1. **The OpenAI Chat Completions schema is the de-facto interop contract**, and it does not formally specify `reasoning_content`. Vendors that need it (DeepSeek, some Qwen variants, soon-to-be others) attach it to assistant messages as an extension field. Clients that round-trip messages through their own normalizer — *every* serious agent does this for compaction, persistence, and replay — silently drop unknown fields by default. The "be liberal in what you accept, conservative in what you send" principle works against round-trip preservation here.

2. **Reasoning-mode contracts are evolving from "informational" to "load-bearing."** A year ago, reasoning content was a UI affordance — surface it to the user, throw it away on the next API call. DeepSeek V4 Pro is the first major model to make it part of the *protocol*: the next request must carry the prior reasoning, or the model refuses to continue the tool-use sequence. This is closer to how function-calling state works (you have to pass tool-call IDs back) than how completion state works (you can summarize and resume freely).

The pattern is identical to the W16 issue with Anthropic's `cache_control` blocks — the same field had to be re-added to round-trip preservation across LiteLLM, opencode, and the Codex Anthropic adapter within the same week, for the same reason: clients normalized it away.

### The falsifiable claim

By **end of W18 (2026-05-03)**, at least one of {opencode, crush, litellm} will ship a named flag — call it `preserve_reasoning_content`, `passthrough_reasoning_content`, or similar — that opts message-history normalizers out of stripping the field. LiteLLM is the most likely first mover because the bug is filed against the proxy layer (where the fix is most centralized), and because the LiteLLM nightly cadence makes shipping a per-provider preservation toggle a one-day turnaround. Crush is second-most likely because its v0.63 cycle is open and the reporter cited a specific provider path. opencode is least likely to ship the flag in W18 — its session compaction is structurally harder to teach a per-field exception to — but most likely to ship it in W19.

If none of the three has a fix in any release channel by end of W18, the workaround will continue to be "use DeepSeek V4 Pro only with the official DeepSeek client SDK," and the OpenAI-compat ecosystem will lose this model in practice for tool-use workflows.

### Operational recommendation

If you are routing DeepSeek V4 Pro through any OpenAI-compatible client today (W17), assume it is broken for any flow that involves a tool call followed by another model turn:

1. **Single-turn, no-tools usage works fine** — the bug is strictly in the round-trip path.
2. **For tool-use flows, route DeepSeek V4 Pro directly through the DeepSeek SDK** until your client ships the preservation flag. The performance hit of bypassing your gateway is smaller than the cost of a 400-loop.
3. **Audit your client for `cache_control`-style preservation work shipped in W16** — the same hooks can usually be reused for `reasoning_content`. If you find a `passthrough_fields` config in your gateway, add `reasoning_content` to it and verify the round-trip.
4. **Subscribe to the three issues above.** The first one to close with a merged PR tells you which client to upgrade first.

This pattern will repeat for every reasoning-mode model that promotes reasoning state from informational to load-bearing. Expect at least one Qwen variant to do the same in W19 or W20.

### The structural cause

Two structural causes:

1. **The IDE surface multiplies environment-shape variance.** A CLI runs in the user's shell, in a directory the user `cd`'d into, with whatever git state that directory has. 2. **MCP-server lifecycle is the new init-system.** [openai/codex#19338] is the headline example: a single config entry starts the same stdio MCP server twice. The cause is that the extension's MCP-config loader runs once per workspace folder, and the dedup key it computes doesn't account for two folders both inheriting a user-scoped MCP config. This is the same shape of bug as systemd duplicating units across `/etc/` and `/usr/lib/`, and the same shape of fix is needed: a canonical-source-of-truth resolution step before spawn. None of the three IDE-embedded agents has this today; all three will need it.

The CLI surface doesn't hit either of these because the CLI runs in *exactly one* directory, with *exactly one* config file resolution, and the user is responsible for the environment. The IDE surface inherits the IDE's environment-shape pluralism and pays the regression cost.

### The falsifiable claim

By **end of W19 (2026-05-10)**, at least one of {Codex VS Code extension, opencode desktop, cline} will ship a release whose changelog explicitly names "MCP server deduplication" or "MCP config resolution" as a fix. Codex is the most likely first mover because [#19338] has a clean reproducer and the codex-rs MCP plumbing is a single owner ([@bolinfest]). opencode's HTTP-API surface will likely add a parallel fix because the same shape of bug is latent there. cline is least likely to ship a named fix in W19 because the cline release cadence is slower and the issue has not been triaged against a specific MCP-related entry point.

A second, narrower prediction: the next Codex stable release (0.125.0 GA, likely W18) **will roll back the unified VS Code extension entry point for stdio MCP servers and gate it behind an opt-in flag**, in the same way that the W17 OOM mitigation in opencode was gated behind `--no-vcs-diff`. The cost of leaving the duplicate-spawn behavior on by default is too high (token spend, race conditions on stateful tools); the cost of breaking users who depend on the new entry point is lower because the alternative is the old path that already worked.

If neither prediction lands by end of W19, the IDE-extension surface is being treated by maintainers as acceptable-tail-of-bugs rather than a structural problem, and the per-merge regression rate will continue at the W17 level into W18-W20.

### Operational recommendation

If you run any of these agents inside an IDE today:

1. **Pin to the last CLI-released version, not the last IDE-released version.** The CLI release is what gets the most-eyeballs testing; the IDE wrapper inherits it later.
2. **Audit your MCP config for inherited entries.** If you have an MCP server defined in your user config and your workspace also opens that folder, you may already be running it twice. Symptoms: doubled tool listings in the agent's `/tools` output, doubled token spend on initial tool discovery, race conditions on stateful tools (filesystem servers especially).
3. **For the Codex VS Code extension specifically**: if you work in any non-git or unborn-git workspace, hold at the pre-0.123 extension version until #19326/#19327 are fixed, or open the workspace via CLI and let the extension attach to the existing session.
4. **For opencode desktop**: the desktop health-check regression in [#24142] is post-1.14.21; if you upgraded for the system-theme fix in 1.14.21, you can stay; if you haven't, hold at 1.14.20 until the next patch.

The IDE-embedded surface is becoming the highest-trust surface for AI coding agents — most users never touch the CLI — but its regression rate is currently 2-3× the CLI surface. This delta is the single biggest predictor of which agent will lose users in W18-W20: the one that lets the IDE-side regression rate stay this high, not the one that ships the most features.


## W17 cross-repo synthesis #7 — provider-shape leakage: per-provider params escape into the wrong adapter (2026-04-24T11:25Z)

### What's visible across ≥2 repos

A class of bug is recurring this week where **a parameter (or message field) shaped for one provider's API is accepted at the gateway/router boundary, then either (a) silently dropped before reaching the right provider, or (b) silently propagated to the wrong provider, or (c) returned in the response despite the caller having explicitly opted out of it**. The common structural cause is that the gateway/router type system collapses heterogeneous provider shapes into a single "loosest-superset" interior shape, then re-projects that interior shape onto whichever provider the route resolves to — without a per-field provenance check.

Four concrete instances landed in W17, in three different upstreams:

- **[BerriAI/litellm#26395]** — DeepSeek V4 Pro `reasoning_content` is stripped from assistant messages before re-send. The interior shape has no slot for `reasoning_content`, so the field is dropped on the inbound normalize step. Symptom: 400 on the next turn.
- **[BerriAI/litellm#26413]** (filed 11:06Z) — The mirror-image polarity. Caller sets `"think": false` to opt out of reasoning; LiteLLM's interior shape doesn't have a `think` toggle, so the flag is dropped on the *outbound* projection to the provider, and the provider returns `reasoning_content` and `thinking_blocks` anyway. Same boundary, opposite direction.
- **[BerriAI/litellm#26400]** — `fix(websearch): deduplicate overlapping keys between kwargs and optional_params`. The websearch route accepts caller-supplied kwargs and provider-defined `optional_params`; when a key appears in both, the merge loses precedence and the wrong value is sent downstream. Same shape as the two above: the interior representation can't tell two provenances of the same key apart.
- **[BerriAI/litellm#26389]** — `WebSearch interception follow-up request ignores custom api_base / api_key, causing 401 and deployment cooldown cascade`. The follow-up request constructs its own client from the interior shape, which has lost the per-deployment `api_base`/`api_key` provenance; the wrong credentials are used; the deployment is cooled down because the 401 is misattributed to it.

A fifth instance, in a different repo: **[anomalyco/opencode#24137]** — "OpenAI-compatible streaming tool calls fail when `function.name` arrives in a later chunk." The interior streaming-tool-call shape assumes name-arrives-first; the provider's stream actually sends it later; the assembler drops the call. Same root: provider-specific stream-shape variance flattened into a single interior shape.

### The structural cause

Every gateway/router that fronts more than one provider faces the same design pressure: the caller wants one API; the providers each want their own; the gateway's job is the translation. There are two implementation disciplines for that translation:

1. **Loosest-superset interior shape.** Define an internal struct with a field for every provider's shape, drop unknown fields on inbound, re-project to the target provider on outbound. Cheap, fast, easy to extend. **Loses per-field provenance.** This is what LiteLLM, opencode's transform layer, and crush's openai-compat provider all do today.
2. **Provenance-tagged interior shape.** Every field carries a `(value, source_provider, was_explicitly_set)` triple; outbound projection consults the triple before sending. Expensive, complex, hard to extend. No agent in our coverage does this.

The cost of the cheap discipline is invisible until two things happen simultaneously: (a) a provider promotes a field from informational to load-bearing (DeepSeek's `reasoning_content`, Anthropic's `cache_control` in W16), and (b) the user mixes that provider with others in the same gateway. Both are happening more often as the provider matrix expands.

The polarity-inversion bug (#26413) is the most diagnostic of the cluster: it shows that the boundary fails *symmetrically*. Drop-on-inbound and project-on-outbound are the same bug viewed from opposite sides of the wire. A fix that only addresses one direction (e.g., "add a `passthrough_reasoning_content` flag" — what synthesis #5 predicted) leaves the other direction broken.

### The falsifiable claim

By **end of W19 (2026-05-10)**, at least one of {LiteLLM, opencode} will ship a config schema for **per-provider passthrough field lists** (something like `provider_specific_fields: { deepseek: [reasoning_content, thinking_blocks], anthropic: [cache_control], openai: [...] }`) rather than a flat global `passthrough_fields` list. The provenance pressure is too high to keep the loosest-superset discipline; the cheapest evolution is to keep it but parameterize the projection by provider.

Two narrower predictions:

- **By end of W18 (2026-05-03)**, LiteLLM #26413 will be triaged as a duplicate / sibling of #26395 and they will be fixed in the same PR, not separately.
- **The webrouter-credential-loss class of bug (#26389)** is structurally identical to #26395 (provenance-loss on the interior shape) and will be fixed only after #26395 is fixed, because the same provenance-tagging refactor is needed for both. If #26395 is patched without a structural fix, #26389 will be re-filed in W18 against a different route (likely the `/v1/responses` passthrough or the file-handler passthrough).

### Operational recommendation

1. **Audit your gateway config for caller-supplied passthrough fields you assume are reaching the provider.** If you have any flag that says "do/don't think," "do/don't cache," "use this `api_base`," verify with a `--debug` request that the value lands at the provider, not just at the gateway.
2. **For tool-use flows that depend on provider-specific message-field round-tripping (DeepSeek, Anthropic with `cache_control`, certain Qwen variants), pin to the provider's official SDK** until your gateway exposes per-provider passthrough lists.
3. **For the streaming-tool-call shape problem (#24137)**: if you route OpenAI-compatible providers through opencode and the provider sends `function.name` in a non-first chunk, your tool calls are silently dropped. Symptom: tool calls appear in logs but don't execute. Workaround: disable streaming for those routes until the assembler is fixed.
4. **Treat any new provider-specific field shipped by a vendor in W18-W20 as a likely round-trip break** in every gateway you use, not just one. The cheapest verification is a two-turn tool-use call against your gateway with the new field set; if turn two 400s, the field is being dropped.

This pattern will recur for every provider-specific field promoted to load-bearing status. The number of such fields across the provider matrix is now ≥ 5 (DeepSeek `reasoning_content`, Anthropic `cache_control`, Anthropic `thinking_blocks`, OpenAI `response_format` strict, Bedrock `guardrailIdentifier`); it will be ≥ 8 by end of Q2.


## W17 cross-repo synthesis #8 — empty-batch bypass: filter→guard→execute pipelines that lose the guard when the filter empties (2026-04-24T11:30Z)

### What's visible across ≥2 repos

A second class of bug is recurring this week, structurally distinct from synthesis #7. The shape: a pipeline of the form `input → filter → guard → execute` is built around the assumption that the guard runs against the filtered set. When the filter happens to empty the set, the guard *also* gets nothing to operate on, and downstream code either (a) executes the unguarded fallback, (b) loops indefinitely on the empty input, or (c) skips a state-recovery step that the guard was supposed to anchor. The bug is invisible until the filter empties.

Four concrete instances landed in W17, in three different upstreams:

- **[charmbracelet/crush#2669]** — "Trapped in an infinite loop when call `WRITE` tool but `input` is empty str." Filter: input-validator. Guard: "if input invalid, surface error to model and let it retry." Execute: write to disk. When input is `""`, the validator empties the action set; the guard's error-surfacing branch is not reached because the dispatch table keys off action presence, not action validity; the agent keeps re-issuing the empty-input call. This is the canonical empty-batch bypass: the *empty set passes every check* because every check is "for each item, validate."
- **[anomalyco/opencode#24137]** (closed) — "OpenAI-compatible streaming tool calls fail when `function.name` arrives in a later chunk." Filter: stream-fragment classifier. Guard: "if a tool call has no name yet, hold the fragment in the buffer." Execute: dispatch the assembled call. When the first fragment arrives without a name, the classifier files it under `unknown`; the guard checks the *known-tool-call* buffer for completeness; the unknown-bucket fragments are dropped at end-of-stream because no guard runs on the unknown bucket. Same shape: the bypass branch (unknown bucket) has no guard.
- **[BerriAI/litellm#26387]** — "BedrockGuardrail streaming hook drops `HTTPException` after HTTP 200, producing broken SSE stream." Filter: response-status check. Guard: "if status >= 400, raise." Execute: write SSE chunks. When the upstream returns 200 but the guardrail hook raises *during* streaming, the status filter has already passed (200), so the exception falls into the "no-error" execute branch and the stream is half-broken. The exception only fires *after* the guard has decided the path is safe.
- **[BerriAI/litellm#26390]** — "Guardrail param handling in list and submission endpoints." Filter: param normalizer. Guard: "if param missing, default to safe value." Execute: enforce guardrail. When the param arrives with the value `None` (vs. absent), the normalizer treats it as set-to-None; the guard sees a set value and skips the default; the executor sees `None` and either crashes or no-ops. The empty case (None vs absent) bypasses the guard.

A fifth instance, narrower: **[anomalyco/opencode#24142]** — "Desktop health check fails during server startup causing IDE instability." The startup sequence is `spawn server → poll health → mount UI`. The poll-health stage has a "if health fails after N retries, abort" guard. When the server is starting and the *first* health probe arrives during a window where the server's HTTP listener is up but the readiness handler is not, the probe returns a 200 with an empty body. The poll's success criterion is HTTP-200 (not body-content); guard is satisfied; UI mounts against a not-actually-ready server; instability cascades. Same shape: the guard's success condition is too loose, so the empty-but-200 case bypasses it.

### The structural cause

The empty-batch bypass is a special case of a more general truth: **most validation and guard code in agent CLIs is written in the affirmative — it checks "do the items I have satisfy condition X" — and not in the negative — it does not check "do I have items at all, and is that itself a problem."** When the design intent is "this guard exists because if no items exist, that's also wrong," the absent-set case has to be *named* in the guard, not implied. Almost no guard names it.

Two ecosystem pressures make this worse this week:

1. **Streaming and incremental dispatch.** Every modern agent has moved to a stream-first or incremental dispatch model where pieces of an action arrive over time. The guard runs once, against the assembled action; until the action is "complete enough" to assemble, the guard literally cannot run. Bugs like #24137 and #26387 are direct consequences: the guard only sees the *complete* set, but execution begins on the *partial* set.
2. **Fallback-and-default chains as a substitute for explicit error paths.** The crush #2669 infinite-loop bug, the litellm #26390 None-vs-absent bug, and the opencode #24142 health-check bug all fall out of the same anti-pattern: rather than raise on the unexpected-empty case, the code falls back to a default (retry the model, default the param, treat 200 as ready). The defaults are individually reasonable; together they remove the place where a hard error would have surfaced the bug.

### The falsifiable claim

By **end of W19 (2026-05-10)**, at least one of {crush, opencode, LiteLLM} will land an explicit `assert_non_empty` or `reject_empty_action` or `require_ready_body` discipline in a code path that previously fell through silently. The most likely first mover is **crush**, because #2669 is a high-visibility infinite-loop bug with a clean reproducer (the model just keeps issuing `WRITE` with empty input until the user kills the process); a "reject empty WRITE input at the dispatcher, return error to model" patch is a one-day fix and shipping it is cheaper than continuing to absorb the issue noise.

A second prediction: **the streaming-incremental-dispatch class of empty-batch bypasses (#24137, #26387) will not get a structural fix in W18 or W19.** The reason is that the structural fix requires moving the guard from "runs once against the assembled action" to "runs incrementally against partial actions, with a notion of safe-prefix" — and no agent in our coverage has that primitive. Instead, expect per-bug patches that hard-code the missing case (e.g., "if first fragment has no name, hold instead of file under unknown"). These patches will work and will not generalize, and the next streaming protocol that ships a new "out-of-order" shape will re-surface the same bug class.

A third, narrower prediction: **the opencode desktop health-check bypass (#24142) will get a fix that tightens the success condition** (likely: require a body containing a `ready: true` field, not just HTTP 200). This is the cheapest patch and the one users will accept; it will land in v1.14.23 or v1.14.24.

### Operational recommendation

1. **Audit your agent for any tool with an empty-input loop risk.** If you use crush or any agent with a similar dispatch shape and have ever seen the model retry the same empty action, set a hard cap on identical-action retries (most agents have a flag for this; if not, run with a per-session iteration cap).
2. **For streaming tool calls through OpenAI-compatible providers**, log the raw stream fragments at debug level for one session and confirm `function.name` arrives in the first chunk. If it doesn't, you are at risk of #24137-shaped silent drops; disable streaming for those routes.
3. **For LiteLLM proxy with Bedrock guardrails on streaming endpoints**, expect SSE corruption when the guardrail rejects mid-stream against a 200-prefixed response. Until #26387 is fixed, either disable streaming on guarded routes or set a client-side timeout that triggers a re-request on incomplete streams.
4. **For opencode desktop**, hold at v1.14.20 or v1.14.21 and skip v1.14.22 until #24142 has a release-tagged fix; the empty-body health-check window is short but real, and the cost of a false-ready mount is the entire IDE session.

The empty-batch bypass class is older than this week — it is a re-instance of the well-known "vacuously true" bug pattern in formal logic, where `∀ x ∈ ∅. P(x)` is always true and quietly satisfies any check. What is new is that agent CLIs are now built almost entirely out of filter-guard-execute pipelines, so the surface area for the bug is now the entire surface area of the agent. Expect at least one new instance in W18, in any repo in our coverage, against any new tool that ships with an input validator.
