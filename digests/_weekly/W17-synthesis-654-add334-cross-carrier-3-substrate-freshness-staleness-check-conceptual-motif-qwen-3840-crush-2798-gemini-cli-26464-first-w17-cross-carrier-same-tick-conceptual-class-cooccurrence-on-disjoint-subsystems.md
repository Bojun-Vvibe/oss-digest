# W17-synthesis-654 — Add.334 cross-carrier 3-substrate "freshness/staleness check" conceptual MOTIF (qwen-code ihubanov #3840 + crush andreynering #2798 + gemini-cli cocosheng-g #26464) emerges as first W17 cross-carrier same-tick conceptual-class co-occurrence on disjoint subsystems

**Tick anchor**: Add.334 (capture window 2026-05-04T22:25:00Z → 2026-05-04T23:15:00Z).

**Founding observation**: Three carriers within the same capture tick produce fresh-opens that share a **conceptual primitive** ("check whether on-disk/cached state has been mutated since last read; refuse / refresh if so") despite implementing it on **structurally disjoint subsystems**:

1. **QwenLM/qwen-code #3840** (ihubanov, FRESH-OPEN 21:04:26Z, head `c6de8c171be7dc9905ffc2ea60b65a04411e3e42`): `feat(core): refuse Edit/WriteFile when the file changed since last read` — file-system staleness guard on tool-execution path. The Edit/WriteFile tool refuses to apply when the on-disk file mtime/hash has changed since the agent's last Read.

2. **charmbracelet/crush #2798** (andreynering, FRESH-OPEN 20:54:45Z, head `defa17365c955a754a6dd30fe52277e18f782b22`): `fix(config): check config file for newer token before OAuth refresh` — credential-cache staleness check on auth-refresh path. The OAuth refresh routine reads the on-disk config first to detect a newer token written by another process before issuing a refresh.

3. **google-gemini/gemini-cli #26464** (cocosheng-g, FRESH-OPEN 20:35:24Z, head `f5a916c38eb187f1954ab797d9a6663df75ab3a6`): `fix(config): ensure configuration persistence and fix in-memory regressions` — in-memory vs persisted state coherence check. In-memory config can drift from persisted config; PR fixes regressions where persistence was bypassed.

**Why this is a MOTIF, not a coincidence**: The three PRs target three completely disjoint subsystems — (1) tool-execution file I/O, (2) auth-token refresh, (3) configuration persistence. They share **no shared dependency, no shared author, no shared upstream issue**. The conceptual unification is at the **invariant-design level**: each PR adds a "before mutating shared state X, verify nobody else mutated X since we last looked" guard. This is a TOCTOU-class invariant being introduced **simultaneously** across 3 carriers on 3 different state-classes (filesystem, credentials, in-memory config). The 3-carrier co-occurrence within a 29m02s window (20:35:24Z → 21:04:26Z) is the first such conceptual-class same-tick MOTIF observed in W17.

**Structural distinction from prior cross-carrier clusters**: Prior cross-carrier clusters in W17 (e.g., synth-572 cross-carrier anchor regime asymmetry; MCP-elicitations DOUBLET in Add.332 codex #21062 + goose #8999) were **substrate-class** clusters: same subsystem (MCP, anchors) on different carriers. synth-654 introduces a **conceptual-class** cluster: same invariant-pattern on different subsystems on different carriers. This is a strictly stronger / more abstract clustering primitive — substrate-class clusters can be explained by upstream protocol releases (MCP spec update propagating); conceptual-class clusters cannot, since there is no shared spec for "freshness checks across heterogeneous state classes". The MOTIF therefore points to **convergent design pressure** in the AI-coding-agent ecosystem rather than spec-propagation.

**Hypothesized cause** (testable, not asserted): AI-coding-agent ecosystem has matured to the point where multiple independently-developed agents are **simultaneously hitting the same class of multi-actor concurrency bug**: agent + user + external process all writing to the same shared state. Each carrier's contributors are independently arriving at TOCTOU-style guards. If this hypothesis holds, expect **continued same-class fresh-opens on additional carriers** (codex, opencode, litellm, goose) within 1–2 W17 ticks targeting freshness-check invariants on yet-different state classes (e.g., MCP server-list, plugin manifest, model-info cache).

If realized, the conceptual-class MOTIF is structurally robust and synth-654 extends to a **4-substrate cross-carrier conceptual-class cluster** with stronger inference about ecosystem-wide convergent invariant-design. **Adjacent / non-instance evidence**: Note that **codex #21055** (aaronl-openai, head `156b1fd1e0bde4bf0c11f0044f658e2deaea30e5` — `Preserve session MCP config on refresh`) is **conceptually adjacent** to the MOTIF (preserve-on-refresh is dual to refuse-on-stale) but **inverse-polarity**: it preserves rather than refuses/refreshes.
