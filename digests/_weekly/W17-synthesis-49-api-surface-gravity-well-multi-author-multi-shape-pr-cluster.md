**Pattern.** A single REST surface (here `/team/*` in litellm) becomes a **gravity well** within one short tick: in the 82 minutes from 00:11Z to 01:33Z, the surface accumulates **5 distinct PRs from 4 distinct authors covering 4 distinct shapes** — feature add (#26455/#26462/#26465 same author, scope-split), hardening (#26464), guardrails composition (#26466), and key-management endpoint (#26468). The PRs do not link each other; their authors do not coordinate in the bodies; the surface is being modified concurrently by people who appear to be discovering each other's work only at PR-list time.

The cross-repo comparison (codex #19458 ChatGPT Library hooks; codex #19456 remote plugin uninstall API) shows that **other coding-agent surfaces in the same window are receiving activity, but single-author and single-shape**. The distinguishing feature of #49 is **multi-author + multi-shape convergence on one surface in one tick**, not just "this surface is busy."

**Why this is distinct from synthesis #45 (concurrent-author scope-split duplicate-feature PR).** #45 names the case of **two PRs trying to ship the same feature** under different scopes, one obsoleting the other. #49 names the case of **N PRs of N different shapes converging on the same surface**: feature, hardening, guardrails, key-management. The #45 pattern is a *pair*; the #49 pattern is a *cluster*. The litellm `/team/*` cluster contains a #45 instance internally (#26455 vs #26462/#26465), but the cluster itself is bigger than #45 can describe.

**Why this is distinct from synthesis #46 (infrastructure shipped as a train of small PRs).** #46 names a **single author or single team** breaking one large change into a series of small PRs by design. #49 names **uncoordinated convergence**: four different authors with four different motivations land on the same URL prefix in the same tick, without any of them having planned the convergence. The signature is that no PR body in the cluster references any other PR body in the cluster.

**Why this is distinct from synthesis #44 (vertical-slice PR stack refreshed in lockstep).** #44 requires lockstep refreshes — the PRs move together on the timeline because they share dependencies. The `/team/*` cluster moves **out of step** — each PR refreshes on its own author's cadence; #26464 merges before #26466 even opens; #26465 reopens before #26464 merges. There is no shared dependency edge.

**Why this matters.**

1. **Merge-order ambiguity becomes acute.** With 5 PRs touching adjacent (and in some cases overlapping) `/team/*` handlers, the order in which they merge determines the eventual semantics. #26464 merged first (00:57Z) before #26465 (refile of budgets feature) had a chance to be reviewed against the hardened metadata path. Reviewers of #26465 must now re-validate against the new post-#26464 baseline, but the PR body still references the pre-#26464 surface.
2. **Cross-PR test coverage is implicit.** None of the PRs adds tests that exercise *another* PR's change. #26466 (guardrails auto-apply at team level) and #26464 (team metadata hardening) interact: a hardened metadata path may reject guardrail attachments that #26466 expects to succeed. No test in either PR covers the interaction.
3. **The `[WIP]` prefix on #26468 is the only social signal in the cluster.** The `[WIP]` convention is used by some litellm authors (#26372, #26373, #26374, #26468 all use it; no other open litellm PR in the window does). It is **not enforced** — reviewers have no automated routing based on the prefix; it functions as an informal "skip me" tag. In the gravity-well context, the absence of `[WIP]` on the other 4 cluster PRs implies they are review-ready, but `[WIP]` on #26468 implies it is not — yet the bulk-key-updates surface in #26468 is the one most likely to interact with #26464 (team key writes) and #26465 (per-model team-member budgets indexed by key).
4. **A new contributor (`Michael-RZ-Berri`, #26468) lands in the busiest spot.** The first-time-this-tick author opens their PR onto a surface that already has 4 concurrent PRs. They cannot reasonably know this; the GitHub PR-creation flow does not surface "this URL prefix has 4 open PRs." Their PR will be reviewed in a context they did not know they were entering.

**Counter-pattern.** A surface with this many concurrent PRs from this many authors in this short a window would, in a coordinated org, route through a **single owning author or working group** who batches the changes. The observation that litellm `/team/*` does not have such routing right now is itself the signal — the surface is hot enough to attract uncoordinated work, hot enough to need coordination, and not yet coordinated.

**Lookback hooks.** Builds on #45 (concurrent scope-split duplicates — present here as #26455/#26462/#26465), #43 (close-and-refile — present here as #26462→#26465), #46 (planned PR train — explicitly *not* what this is), and #44 (vertical-slice lockstep — explicitly *not* what this is). #49 names what happens when the social shapes #43, #45, and several uncoordinated `fix:`/`feat:` shapes all happen on **one URL prefix in one tick**.

**Predictive shape.** Within the next few ticks, expect either (a) a maintainer comment on one of the cluster PRs asking authors to coordinate, (b) a merge of one cluster PR that forces non-trivial rebases on the other 4, or (c) a new PR with a title like *"refactor(teams): consolidate /team/* handlers"* that implicitly acknowledges the surface has become unmanageable.
