The headline is a security hardening proposal in [#4918](https://github.com/Aider-AI/aider/pull/4918) by Apoorv13, which adds a warning when a workspace-level `.aider.conf.yml` contains shell-executing keys (`lint-cmd`, `test-cmd`, `notifications-command`). The motivating attack is concrete: because `auto-lint: true` is the default, a malicious repo can ship a config that silently runs arbitrary shell commands as soon as the user edits code. Anyone who routinely opens untrusted repos in Aider should track this one — it's currently just a warning rather than a prompt/opt-in, so the design discussion is worth following before it lands.

The other two PRs, both from biefan, are narrow robustness fixes to repo map generation:

- [#4920](https://github.com/Aider-AI/aider/pull/4920) catches `tree_sitter.QueryError` in `RepoMap.get_tags_raw()` so a single `.scm` query that's out of sync with the installed parser no longer aborts the whole repo map. It mirrors the existing handling of parser-init failures.
- [#4921](https://github.com/Aider-AI/aider/pull/4921) fixes `get_ranked_tags()` to stop assuming a fixed `(src, dst, data)` tuple shape from `MultiDiGraph.out_edges(..., data=True)`. In environments where the multiedge view includes an edge key, the current unpacking raises `ValueError` and kills repo map generation.

Both are the kind of defensive patches that explain otherwise mysterious "repo map crashed" reports, so reviewers familiar with the tree-sitter and networkx code paths could likely fast-track them. No follow-up issues were filed alongside, and there is no release activity to flag.
