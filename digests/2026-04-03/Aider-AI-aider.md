[#4987](https://github.com/Aider-AI/aider/pull/4987) ("fix: sanitize surrogate characters in messages before sending to LLM") by graydeon targets a long-standing Windows bug tracked in issue #3460. On certain Windows locales (notably Chinese editions), file contents or console input can contain lone UTF-16 surrogate code points such as `\udcb0`. When httpx later tries to JSON-encode the outgoing request body, those surrogates trigger a `UnicodeEncodeError` and the LLM call fails outright. The PR introduces a `sanitize_for_utf8()` helper that recursively walks the message structure (handling nested dicts/lists, not just top-level strings) and replaces any surrogate code points with the Unicode replacement character `U+FFFD`. The sanitiser is invoked inside `send_completion()` immediately before the messages are handed to `litellm.completion()`, which keeps the fix narrowly scoped to the network boundary rather than mutating message history elsewhere in the app.

The change is small, defensive, and unlikely to affect non-Windows users, but reviewers will probably want to confirm: (a) that lossy replacement is acceptable rather than, say, surrogateescape round-tripping back to bytes; (b) that tool/function-call payloads and image parts are also traversed; and (c) that there's a regression test exercising a message containing `\udcb0`. Given how often this class of error has been reported against aider on Windows, this is the one item from today's window worth opening if you triage Windows issues or maintain the LLM transport path.
